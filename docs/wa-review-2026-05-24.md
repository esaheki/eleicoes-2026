## Summary

Event-driven serverless pipeline collecting social-media sentiment for the 2026 Brazilian presidential election. CDK IaC is thorough, TTL hygiene is excellent, KMS encryption is active on Kinesis, and the DLQ + bisect pattern on the Processor Kinesis event source is a genuine strength. Main gaps are around operational resilience for election-night traffic (no CI/CD, missing WAF, one unprotected DLQ path in the Broadcaster) and a handful of cost/observability issues.

## High Risk Issues

### HRI-1 â€” Reddit credentials leak into CloudFormation template (Security)
**Status: OPEN**

`streaming-stack.ts:149-150` â€” `REDDIT_CLIENT_ID` and `REDDIT_CLIENT_SECRET` are injected as plaintext Lambda env vars from the deploying machine's shell environment. All other API keys use SSM paths fetched at runtime. These end up visible to anyone with `cloudformation:GetTemplate`.

**Fix:** Add `REDDIT_CLIENT_ID_SSM: '/eleicoes2026/reddit-client-id'` and `REDDIT_CLIENT_SECRET_SSM: '/eleicoes2026/reddit-client-secret'`, fetch at runtime in `packages/collector/src/secrets.ts`. SSM path already covered by the existing wildcard IAM grant.

**Effort:** Low

### HRI-2 â€” No WAF on API Gateway or CloudFront (Security)
**Status: RESOLVED** â€” committed 2026-05-24

## Medium Risk Issues

### MRI-1 â€” No CI/CD pipeline; all deployments are manual (Operational Excellence)
**Status: OPEN**

Manual `npx cdk deploy` + `aws s3 sync` + CloudFront invalidation is the entire deploy process. No rollback gate, no typecheck/build gate, no diff preview.

**Fix:** GitHub Actions workflow: typecheck â†’ build gate on PR, `cdk diff` as check, `cdk deploy` on merge to `main`.

**Effort:** Medium

---

### MRI-2 â€” No AWS X-Ray tracing (Operational Excellence)
**Status: OPEN**

No Lambda or API Gateway has active tracing. Latency bottlenecks across Comprehend â†’ Bedrock â†’ DynamoDB chain are opaque.

**Fix:** Add `tracing: lambda.Tracing.ACTIVE` to all `NodejsFunction` constructs. Add `tracingEnabled: true` to REST API `deployOptions`.

**Effort:** Low

---

### MRI-3 â€” PITR only on `election-sentiment`; other tables unprotected (Reliability)
**Status: OPEN**

`misinfoEventsTable` (30-day forensic log) and `misinfoAggregatesTable` lack `pointInTimeRecovery: true`. A bad deploy or accidental delete destroys irreplaceable misinfo data.

**Fix:** Add `pointInTimeRecovery: true` to `misinfoEventsTable` and `misinfoAggregatesTable`.

**Effort:** Low

---

### MRI-4 â€” Broadcaster calls `getConnections()` once per score update in a batch (Performance)
**Status: OPEN**

`packages/broadcaster/src/index.ts:141-143` â€” `broadcast()` runs a full paginated DynamoDB Scan on `ws-connections` for every score update in the batch. 4 candidates updating simultaneously = 4 Scans against the same table.

**Fix:** Hoist `getConnections()` above the loop and pass the pre-fetched list into `broadcast()`.

**Effort:** Low

---

### MRI-5 â€” No CloudWatch log retention policy (Cost)
**Status: OPEN**

All Lambda log groups default to "Never expire". At election-day volume this accumulates significant storage cost and slows log searches.

**Fix:** Add `logRetention: RetentionDays.ONE_MONTH` (or `TWO_WEEKS`) to all `NodejsFunction` constructs.

**Effort:** Low

---

### MRI-6 â€” `PRICE_CLASS_ALL` on both CloudFront distributions (Cost)
**Status: OPEN**

`website-stack.ts:101` and `website-stack.ts:131`. `PRICE_CLASS_200` covers South America, North America, Europe, and Asia â€” sufficient for this workload and ~30% cheaper per GB transfer.

**Fix:** Change both to `cloudfront.PriceClass.PRICE_CLASS_200`.

**Effort:** Low

## Improvement Opportunities

### IMP-1 â€” Automate YouTube quota disable via SNS â†’ Lambda (Reliability)
**Status: OPEN**

`YouTubeQuotaAlarm` sends SNS email but requires a human to write `disabled_until` to `collector-state`. On election day this manual step may not happen fast enough to prevent a 24h API ban.

**Fix:** Create a `quotaDisablerLambda` subscribed to `alertTopic` that auto-writes `disabled_until = midnight BRT`. Pair with an EventBridge rule at 00:05 BRT to clear the flag.

**Effort:** Medium

---

### IMP-2 â€” WebSocket has no connection authentication (Security)
**Status: OPEN**

`wsConnectLambda` registers any connection without validation. Bots could saturate the `ws-connections` table and inflate broadcaster fan-out cost.

**Fix:** Accept an `?token` query parameter on connect, validated with HMAC signed by a shared secret from SSM. Return HTTP 401 for missing/invalid tokens.

**Effort:** Medium

---

### IMP-3 â€” No Provisioned Concurrency for Processor and API Lambdas on election day (Performance)
**Status: OPEN**

Cold starts for the 1024 MB Processor Lambda (Bedrock + Comprehend SDK init) and API Lambda will cause P99 spikes when traffic surges after polls close.

**Fix:** Scheduled Application Auto Scaling: pre-warm Processor to 2 and API Lambda to 5 concurrent instances starting 30 min before polls close (21:00 BRT). Disable after midnight.

**Effort:** Medium

---

### IMP-4 â€” `apifyCollectorLambda` can have overlapping invocations (Reliability)
**Status: OPEN**

300s timeout + 5min schedule = potential overlap if Apify is slow. Causes duplicate posts to Kinesis.

**Fix:** Add `reservedConcurrentExecutions: 1` to `apifyCollectorLambda`. Add a `Throttles` metric alarm to detect chronic slowness.

**Effort:** Low

---

### IMP-5 â€” No S3 lifecycle policy on PipelineStack raw data (Sustainability)
**Status: OPEN**

Firehose â†’ S3 â†’ Glue data accumulates indefinitely in Standard storage post-election.

**Fix:** S3 lifecycle rule: Intelligent-Tiering after 30 days, Glacier Instant Retrieval after 90 days, expire or Deep Archive after 365 days.

**Effort:** Low
